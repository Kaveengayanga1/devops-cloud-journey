Troubles observed

- Pod remained in Init:0/1 and nginx container stayed Waiting (PodInitializing).
- sidecar init container repeatedly logged "No such file or directory" for /var/log/nginx/access.log and /var/log/nginx/error.log.

Root cause analysis

- The sidecar was defined under initContainers with a command that runs an infinite loop (while true; do ...; sleep 30; done). Traditional init containers must complete (exit) before normal containers start. An infinite-loop init container prevents normal containers from starting.
- Because the sidecar init container never completed, the nginx container could not start, so it never created the log files in the shared volume. The sidecar therefore repeatedly reported "No such file or directory".

Mistakes made

- Using an infinite-loop sidecar as an initContainer. Init containers are intended for one-time setup or blocking tasks that complete, not long-running daemons.
- Expecting log files to already exist when the sidecar started (the log files are generated by nginx when it starts).

Fix applied (as performed in the exercise)

- Delete the problematic Pod: `kubectl delete pod webserver`.
- Update behavior so the sidecar runs as a long-running process while allowing nginx to start. Two approaches:
  1. Move the sidecar into `containers` (make it a regular container) so it and nginx run concurrently and share the emptyDir.
  2. If using Kubernetes 1.28+ and Native Sidecar support, mark the initContainer with `restartPolicy: Always` inside the initContainers section to indicate a sidecar-like behavior so kubelet will start regular containers while sidecar keeps running. In the exercise the latter was used (added `restartPolicy: Always`).

Corrected manifest excerpt (as used in the successful run)

apiVersion: v1
kind: Pod
metadata:
name: webserver
labels:
app: webserver
spec:
volumes: - name: shared-logs
emptyDir: {}

initContainers: - name: sidecar-container

Notes and recommendations (professional)

- Preferred pattern: run sidecar as a regular container (not initContainer) so it starts concurrently with the main container. That avoids init semantics confusion and ensures sidecar and main container lifecycle are aligned.
- If init-like ordering is absolutely required, use a small init container that performs finite setup tasks (e.g., create directories, copy baseline files), exit successfully, and then let main containers start.
- For Kubernetes 1.28+ native sidecar semantics can be used; test cluster version compatibility before relying on `restartPolicy: Always` under initContainers.
- Ensure log file paths and ownership are compatible: some images write logs to different locations or require specific permissions.
- For production consider using a lightweight log forwarder (fluent-bit/rsyslog/filebeat) as a sidecar and rotate logs or send to central logging; avoid blocking I/O operations inside busy loops.
